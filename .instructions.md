# Harmonic Coach Agent Instructions

## Product intent

Harmonic Coach is an AI lead-sheet tutor for piano students. The product helps users understand chord charts measure by measure through clear harmonic analysis, instructor-style explanations, and eventually browser-based piano demonstrations.

The goal is not to build a generic chatbot. The goal is to build an inspectable, interactive learning interface for music theory.

## Engineering principles

- Use TypeScript throughout.
- Prefer simple, explicit code over clever abstractions.
- Keep components small and readable.
- Avoid adding dependencies unless they clearly improve the product.
- Keep domain logic in `src/lib`.
- Keep reusable UI in `src/components`.
- Avoid mixing music-theory logic directly into presentation components.

## AI product principles

- Treat AI output as uncertain unless validated.
- Prefer structured outputs over freeform blobs.
- Show confidence and alternative interpretations when analysis is ambiguous.
- Use deterministic parsing and analysis where possible before relying on LLM explanations.
- The UI should help users understand why an analysis was produced.

## Code quality principles

- Prefer small, focused modules with one clear responsibility.
- Keep React components primarily responsible for rendering and interaction.
- Keep domain logic, parsing, and analysis functions in src/lib.
- Use clear, intention-revealing names for variables, functions, components, and types.
- Avoid vague names like data, item, thing, helper, or utils when a domain-specific name is available.
- Avoid deeply nested conditionals. Prefer early returns, small helper functions, or lookup maps when they improve readability.
- Do not introduce abstractions before there is a repeated pattern worth abstracting.
- Do not create large catch-all files. Split code by product/domain responsibility.
- Prefer pure functions for chord parsing, harmonic analysis, formatting, and validation.
- Keep component props explicit and typed.
- Avoid any unless there is a clear boundary with unknown external data.
- Do not silence TypeScript or ESLint errors without explaining why.
- Remove unused code, unused imports, dead comments, and placeholder boilerplate.
- Comments should explain non-obvious product, domain, or music-theory decisions. Do not comment obvious syntax.
- Optimize first for correctness, clarity, and extensibility; optimize for cleverness never.

## Current scope

V0 includes:

- Lead-sheet chord grid with editable measures
- Selected-measure harmonic analysis panel
- AI-powered harmonic analysis via Anthropic Claude
- Session-based workspace (visible after initial editing/analysis)
- Measure timeline with harmonic function indicators
- Sticky timeline visualization
- Responsive, warm, polished UI (desktop + mobile)
- Runtime validation of AI-generated analysis

## Out of scope for V0

Do not add these yet:

- Authentication
- Database
- PDF/image ingestion
- Audio playback
- MusicXML import/export
- Payments
- User accounts
- Complex routing
- Overbuilt state management
- Browser-based piano demonstrations
- Alternative analysis interpretations UI
- User-specific settings persistence

## Design direction

The app should feel warm, musical, focused, and premium. Avoid generic SaaS styling. Favor spacious layouts, strong typography, rounded panels, subtle borders, and clear interaction states.

Session workspace is hidden on landing; users see a hero message and song setup. Workspace (lead sheet, analysis panel, timeline) appears only after the first edit or analysis request, keeping the initial experience clean and focused.
